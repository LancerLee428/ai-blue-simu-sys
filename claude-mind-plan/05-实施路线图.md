# 实施路线图

## 总体策略

采用**迭代增量**方式，优先实现核心功能，再逐步增强视觉效果。每个迭代都交付可演示的功能增量。

## 阶段划分

### 第一阶段：武器系统基础（P0，核心功能）

**目标**：实现导弹发射、飞行、命中的完整流程

**工作量估算**：中等

**任务清单**：

1. **类型系统扩展**
   - [ ] 在 `types/tactical-scenario.ts` 中添加武器相关类型
   - [ ] 定义 `WeaponType`, `GuidanceType`, `WeaponSpec`, `Weapon`
   - [ ] 定义 `WeaponLaunchEvent`, `WeaponImpactEvent`

2. **武器数据库**
   - [ ] 创建 `services/weapon-database.ts`
   - [ ] 录入常见武器数据（空空导弹、防空导弹、反舰导弹）
   - [ ] 至少包含 10 种武器型号

3. **武器系统核心**
   - [ ] 创建 `services/weapon-system.ts`
   - [ ] 实现 `launch()` 方法
   - [ ] 实现 `update()` 方法（物理更新）
   - [ ] 实现比例导引法制导算法
   - [ ] 实现命中判定逻辑

4. **集成到推演引擎**
   - [ ] 在 `ExecutionEngine` 中集成 `WeaponSystem`
   - [ ] 在 `runAnimationLoop()` 中调用武器更新
   - [ ] 连接武器事件回调

5. **基础渲染**
   - [ ] 在 `MapRenderer` 中添加 `renderWeapons()` 方法
   - [ ] 使用 Billboard 渲染导弹实体
   - [ ] 使用 Polyline 渲染导弹轨迹

**验收标准**：
- ✅ 能够通过 AI 生成的战术方案触发导弹发射
- ✅ 导弹能够追踪目标并飞行
- ✅ 导弹命中时触发事件
- ✅ 在地图上能看到导弹飞行轨迹

## 实施建议

### 开发顺序
1. **先纵向，后横向**：先完成一个完整的武器发射-飞行-命中-爆炸流程，再扩展更多武器类型
2. **先功能，后效果**：先保证逻辑正确，再优化视觉效果
3. **先单元，后集成**：每个服务独立开发和测试，最后集成到推演引擎

### 测试策略
1. **单元测试**：为 `WeaponSystem`, `DamageCalculator` 编写单元测试
2. **集成测试**：创建测试场景，验证完整流程
3. **性能测试**：压力测试（100+ 导弹同时飞行）

### 风险控制
1. **性能风险**：粒子系统可能影响帧率 → 提前实现 LOD 和池化
2. **复杂度风险**：物理模拟可能过于复杂 → 先用简化模型，后续迭代
3. **兼容性风险**：Cesium 版本差异 → 锁定 Cesium 版本

### 里程碑
- **M1（2 周）**：完成第一阶段，可演示导弹发射和飞行
- **M2（3 周）**：完成第二、三阶段，可演示完整战斗流程
- **M3（4 周）**：完成第四、五、六阶段，达到专业水准
- **M4（5 周）**：完成第七阶段，生产就绪

## 总结

本设计方案提供了一个完整的、可落地的实施路线，从核心功能到视觉增强，从基础实现到性能优化，覆盖了态势效果扩充的全部需求。

**关键成功因素**：
1. 严格遵循 RAF 动画规范（禁用 setTimeout/setInterval）
2. 保持单向数据流（Engine → Store → UI）
3. 及早进行性能测试和优化
4. 每个阶段都交付可演示的增量

**预期成果**：
- 专业的武器系统（导弹发射、飞行、命中）
- 真实的爆炸效果（火球、冲击波、烟雾）
- 动态的传感器效果（雷达扫描、电子战波纹）
- 完整的战斗逻辑（损伤评估、战损统计）
- 流畅的性能表现（30+ FPS）
