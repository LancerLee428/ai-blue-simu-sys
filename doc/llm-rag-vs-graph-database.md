# LLM + RAG 与图数据库能力边界对比

## 一、文档目的

本文用于单独说明 `LLM + RAG` 与图数据库在本项目中的职责边界、适用场景和替代关系，避免在架构讨论中将“语义理解能力”和“结构化关系建模能力”混为一谈。

核心结论只有一句话：

> **图数据库不是当前阶段的必选项，但结构化关系建模是必选项；`LLM + RAG` 可以增强关系数据的理解和访问方式，但不能替代关系建模本身。**

---

## 二、要先区分的三个问题

在讨论是否需要图数据库时，实际上混杂了三个不同层次的问题：

### 1. 是否需要关系建模

答案是：**必须需要**。

因为系统天然存在大量对象关系，例如：

- 兵力单元与指挥层级之间的隶属关系
- 平台与装备之间的装备关系
- 单位与区域之间的部署关系
- 单位与任务之间的执行关系
- 对象与事件之间的参与关系
- 对象在不同想定、不同时间点下的动态关系

这些关系不是模糊语义，而是系统事实的一部分，必须结构化表达。

### 2. 是否需要图数据库

答案是：**不一定**。

如果当前需求主要是：

- 对象增删改查
- 基础关系维护
- 一跳或两跳关联查询
- 想定展示与态势可视化
- 基础分析与筛选

那么 `PostgreSQL + PostGIS` 就足以承载。

只有当后续出现以下场景时，图数据库价值才会明显上升：

- 高频多跳关系遍历
- 路径发现与影响链分析
- 指挥链、补给链、威胁传播链分析
- 复杂图查询性能瓶颈
- 专门的图算法需求

### 3. `LLM + RAG` 能否替代前两者

答案是：

- **不能替代关系建模**
- **不能完整替代图数据库**
- **可以替代一部分“关系访问入口”和“关系解释层”**

这就是本系统需要把握的能力边界。

---

## 三、为什么 `LLM + RAG` 不能替代关系建模

### 1. `RAG` 的本质不是事实存储，而是检索增强生成

`RAG` 擅长的事情是：

- 从文档中召回相关片段
- 基于上下文生成自然语言回答
- 把用户问题映射为更容易理解的知识表达
- 对已有结果做总结、解释和归纳

但关系建模要求的是：

- 确定性
- 可验证性
- 可审计性
- 可重复查询
- 可精确更新
- 可被业务规则、仿真逻辑和分析流程稳定消费

两者不是同一类能力。

### 2. 没有显式关系建模时，系统得到的是“推断关系”，不是“受控关系”

如果对象之间的关联不通过对象表、关系表、事件表等结构化方式记录，而是依赖：

- 文档说明
- 报告片段
- Embedding 召回
- LLM 归纳总结

那么系统最终得到的大多是“推断出来的关系”，而不是“系统显式维护的关系”。

对于通用问答类系统，这种方式可能足够；但对于本项目这类需要强事实约束的平台来说，风险很大。

### 3. 时态关系与场景关系不能靠纯语义检索稳定支撑

本项目关系数据不是静态知识，而是与以下因素强相关：

- 想定上下文
- 推演时间
- 状态变化
- 生效/失效区间
- 触发事件来源

例如：

- 某单位是否在当前想定中隶属于某指挥链
- 某平台在某时间点是否部署于某区域
- 某关系是全局关系还是仅在某个场景内成立

这类问题需要精确的结构化关系和时间语义支持，不能依赖 `LLM + RAG` 的概率式生成来承担事实真相。

### 4. 缺乏可审计性

如果系统需要回答：

- 为什么判定 A 与 B 存在关系？
- 这条关系从何而来？
- 是谁建立的？什么时候生效的？
- 这条关系当前是否仍然有效？

那么系统应当能够返回：

- 关系记录来源
- 关系类型
- 关系 ID
- 生效时间与失效时间
- 关联对象 ID
- 触发事件或人工录入依据

而不是仅返回一句“根据召回内容推测相关”。

---

## 四、`LLM + RAG` 真正适合承担什么角色

`LLM + RAG` 在本系统中并不是没有价值，相反，它非常适合承担以下角色。

### 1. 自然语言访问入口

用户未必会构造复杂查询，但可以自然地提问，例如：

- 当前想定下有哪些蓝方平台部署在东南区域？
- 与某高风险目标存在直接或间接关系的对象有哪些？
- 最近 24 小时有哪些关键态势变化？

`LLM` 可以把这些自然语言问题转译为：

- SQL 查询条件
- 关系遍历参数
- 对象筛选逻辑
- 联合检索策略

### 2. 结构化结果解释层

当结构化关系查询已经查出结果后，`LLM` 很适合做：

- 结果摘要
- 路径解释
- 原因说明
- 面向非技术用户的文字化表达

也就是说，`LLM` 很适合“解释查出来的关系”，而不是“替代关系本身”。

### 3. 文档知识与结构化数据的联合分析层

本项目并不只有数据库中的对象与关系，还可能存在：

- 条令文档
- 研究报告
- 战法说明
- 案例材料
- 外部情报文本

`RAG` 的价值在于：

- 从文档中补充背景知识
- 将文本知识与结构化对象关系联合分析
- 形成更接近专家表达方式的解释结果

### 4. 查询辅助与分析辅助

`LLM` 还适合承担：

- 查询意图补全
- 分析路径推荐
- 相关对象与关系的辅助提示
- 报告草稿与解释性文本生成

这些能力都能显著改善系统易用性。

## 五、图数据库与 `LLM + RAG` 的本质差异

| 维度 | 图数据库 | `LLM + RAG` |
|---|---|---|
| 核心作用 | 存储与计算结构化关系 | 理解、检索、解释和生成语义内容 |
| 事实精度 | 高，关系显式且可验证 | 概率式，依赖召回与生成 |
| 多跳关系遍历 | 强 | 弱，不稳定 |
| 路径分析 | 强 | 不擅长 |
| 审计追踪 | 强，可返回明确来源 | 弱，更多是语言层解释 |
| 时间/场景关系表达 | 可通过结构建模精确表达 | 难以单独稳定承担 |
| 用户交互友好度 | 一般，需要专业查询方式 | 强，自然语言友好 |
| 适合作为系统真相来源 | 是 | 否 |
| 适合作为解释层 | 一般 | 是 |
| 适合作为语义入口 | 一般 | 是 |

这张表说明了一个核心事实：

> **图数据库与 `LLM + RAG` 不是同类替代关系，更像是“结构化底座”与“语义访问层”的分工关系。**

---

## 六、本项目推荐的能力分层

对于本项目，更合理的分层方式是：

### 1. 关系建模层

当前由 `PostgreSQL + PostGIS` 承担，负责：

- 对象主表
- 对象关系表
- 空间数据表
- 想定上下文
- 状态变更记录
- 事件记录

这是当前阶段的事实底座。

### 2. 图分析层

当前阶段可以暂不引入专门图数据库。

未来如果出现以下需求，再考虑引入图数据库或图分析引擎作为增强层：

- 高频多跳关系分析
- 复杂路径发现
- 威胁链、指挥链、影响链分析
- 图算法需求
- PostgreSQL 递归查询性能或表达能力成为瓶颈

即使未来引入图数据库，也建议其定位为：

- 分析增强层
- 查询加速层
- 图计算层

而不是系统唯一主事实源。

### 3. AI 语义层

`LLM + RAG` 应定位为：

- 自然语言访问入口
- 文档语义检索层
- 结构化结果解释层
- 辅助分析层

它的价值在于“让用户更容易消费结构化关系”，而不是“取消结构化关系建模”。

---

## 七、什么时候图数据库可能真正值得引入

图数据库并非后期一定要引入，但当出现以下信号时，应认真考虑：

1. **多跳关系查询成为高频场景**  
   不再只是查一跳、两跳关系，而是频繁进行复杂遍历。

2. **路径分析成为核心能力**  
   需要分析指挥链、补给链、影响链、威胁传播链等路径问题。

3. **递归 SQL 复杂度和性能成为瓶颈**  
   使用 `PostgreSQL` 递归查询已明显难以维护或难以满足性能目标。

4. **图算法成为分析需求**  
   开始需要中心性、聚类、社区发现、传播分析等能力。

5. **AI 层需要稳定消费复杂图上下文**  
   不再只是对平面结构做解释，而是需要建立稳定的复杂关系图上下文供模型消费。

在这些信号尚未出现前，优先把对象关系建模做好，比提前引入图数据库更重要。

---

## 八、最终结论

### 1. 结构化关系建模是必须的

无论未来是否引入图数据库，本系统都必须显式建模对象、关系、时态和场景上下文。

### 2. 图数据库不是首期必选项，也不一定是后期必选项

是否引入图数据库，应取决于图分析复杂度与性能压力，而不是出于“系统看起来应该有图谱能力”的预设。

### 3. `LLM + RAG` 不能替代关系建模，也不能替代图数据库的确定性计算能力

`LLM + RAG` 更适合作为：

- 语义访问入口
- 文档增强层
- 结构化结果解释层
- 辅助分析层

### 4. 更合理的架构关系是“协同”而不是“替代”

对于本项目，最合理的长期思路是：

- **结构化底座**：`PostgreSQL + PostGIS`（必要时后续补图分析层）
- **语义增强层**：`LLM + RAG`
- **用户交互层**：通过自然语言降低复杂关系数据的使用门槛

也就是说：

> **AI 可以让关系数据更好用，但不能让关系数据不建模。**
